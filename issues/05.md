### Problem
`08`/`08b` join against the Jan-2023 shapefile. A neighbourhood that appears in a future
year but not in that shapefile lands silently in `no_data`. The name-mapping rescue
(`08b`) is the right pattern for drift, but it's a hand-curated 2026 snapshot
(`neighbourhood_name_mappings_20260519.csv`) — new unmapped neighbourhoods in 2027 fall out
until a human curates them.

### Fix
Two parts, both already partly designed:
1. Identify the canonical current boundary source (UAlberta Library data services — open item #12).
2. Make the refresh **surface** new unmapped NA-id neighbourhoods for human review rather than dropping them silently — this is the intended Sanity-Agent gate. `08b` already writes an audit log; wire it so a non-empty "unresolved" set is a visible refresh-blocker, not a quiet CSV.

### Acceptance
- [ ] Decision recorded: keep 2023 shapefile vs adopt a newer canonical source (note vintage in the manifest, #1).
- [ ] `08b` audit log's unresolved count is reported prominently in the refresh run output.
- [ ] Refresh runbook documents the human step: curate new mappings → append to a new dated `neighbourhood_name_mappings_<YYYYMMDD>.csv` (§4.7, never edit in place).

### Notes
This issue is correctly **human-gated, not automatic** — that's by design. The goal is to
make the gate loud and the queue explicit, not to auto-resolve names (§4.7 forbids fuzzy
matching).
