**Label:** `refresh-by-design`, `pipeline`
**CLAUDE.md:** §4.3 (year-invariant), §9 (negative rules)

### Problem
`2026` is baked into output/input filenames across the chain, so a new year needs manual
edits in five files in the right order:

- `01_load_data.R` → writes `assess_2026_no_parking.csv`, `assess_2026_with_flags.csv`
- `06_apply_layer1a_rules.R` → reads both above, writes `assess_2026_clean.csv`
- `07_layer2_aggregates.R` → reads `assess_2026_clean.csv`, writes `neighbourhood_aggregates_2026.csv`
- `08_build_geojson.R` → reads aggregates, writes `neighbourhoods_2026.geojson`
- `08b_name_fallback_rescue.R` → reads/writes the `_2026` names + `neighbourhoods_2026_recovered.geojson`

### Fix
Single source of the period. One `YEAR` (and `CITY`) resolved once — ideally in
`pipeline/_shared/config.R` sourced by every script — and all paths built with
`glue("assess_{YEAR}_clean.csv")`. No bare year literal in any path.

### Acceptance
- [ ] `pipeline/_shared/config.R` (or equivalent) defines `YEAR`/`CITY` in one place.
- [ ] Scripts 01, 06, 07, 08, 08b build every input/output path from that variable.
- [ ] Grep for a 4-digit year literal in `scripts/*.R` returns only the deliberately-frozen oracle `year_tag` (see #3) and comments.
- [ ] Re-running with `YEAR <- 2026` reproduces today's filenames byte-for-identical (proves the refactor is behaviour-preserving).

### Out of scope
Don't touch the validation oracle's `2023` tag — that's frozen by design (#3).
