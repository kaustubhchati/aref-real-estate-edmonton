### Problem
`dataSources.js` is a hardcoded manifest-in-miniature: `YEARS = [2026]` and a
`DATA_SOURCES` table written by hand. The seam is right (the page resolves a URL from
`(city, year)` controls), but adding 2027 needs two synchronized manual edits — append a
`YEARS` entry **and** a `DATA_SOURCES` row — or the year button never appears.

### Fix
Fetch `manifest.json` (#1) and derive `CITIES`, `YEARS`, `DEFAULT_*`, and `resolveDataUrl`
from it. The hand-maintained table goes away; the file becomes a thin reader over the
manifest. Keep it legible (§6) — a learner should still be able to read it top to bottom.

### Acceptance
- [ ] `dataSources.js` (or the page) fetches `/data/manifest.json` once and exposes the same public surface (`CITIES`, `YEARS`, `resolveDataUrl`, `describeEmpty`).
- [ ] No year/city literals remain in the file except as fallbacks.
- [ ] Adding a dataset to the manifest makes its year button + map appear with zero frontend edits (manually verify by adding a fake 2027 entry).
- [ ] Empty-state behaviour preserved for (Calgary, *) and (Edmonton, missing-year).
